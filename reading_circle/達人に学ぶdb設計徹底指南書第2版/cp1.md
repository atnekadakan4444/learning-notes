# 第１章

- 論理設計と物理設計はトレードオフ
- データとは「ある形式にそろえられた事実」
- 正規化は RDB でしか行われない設計手法
- データの管理構造・仕様を DB の「モデル」という
- DB のモデルによっては設計技法が異なってくるが、DBMS の違いが設計技法に影響を与える側面は薄い
- 近年は「DOA」（データ中心アプローチ）が主流
  - プログラムよりも先にデータの設計を先に始める
- 従来は「POA」（プロセス中心アプローチ）で逆が主流だった
- プログラム（処理のプロセス・ロジック）は頻繁に増減や変更があるため、プロセス中心だと DB 側も併せて増減や変更を強いられる
  - よって、DOA は「データがあまり変化しない（永続的）」ということを念頭に「プロセスの変更に柔軟に対応する」スタンス
- DOA を重視する以上、設計がアンチパターンになると致命的にシステムの品質まで影響が及ぶといえる
- **外部スキーマ**
  - ビューの世界、ユーザーが触れる視界
  - ユーザーが触れる UI の部分（入力フォーム、登録内容）が該当
- **概念スキーマ**
  - テーブルの世界、開発者の視界
  - ~~DB に開発者が実装した内容~~ → 論理や構想の段階なので「論理設計」
- **内部スキーマ**
  - DBMS の世界、データの物理的な配置
  - DB に開発者が実装した内容（ファイルとして書き起こしている内容） → ファイルと言う実装した実態が伴うので「物理設計」
- Q. 概念スキーマは必要なの？　 A.外部スキーマと内部スキーマの緩衝材の役割を担う
  - 小規模な開発だと 3 層まで至らず 2 層で事足りる → 大規模になると非効率を招く原因になる
    - 概念スキーマがない = 外部スキーマと内部スキーマの依存関係が強まる = 変更に対する柔軟性が無い
  - 「**データ独立性**」（スキーマの独立性）が柔軟な変更を可能にしている
    - **論理的データ独立性**：外部スキーマからの独立
    - **物理的データ独立性**：内部スキーマからの独立
